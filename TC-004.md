# Resilience - Database Connectivity Interruption

## Implementation Details

---

### 🔧 Pre-conditions

- System is running with Kafka and MongoDB containers.
- `TestContainerManager` supports container pausing via Docker Client.

---

### 🧪 Test Steps

| Step | Area | Description |
|------|------|-------------|
| **Step 1** | MongoDB Setup | Ensures the product **QA-RESILIENCE** exists with stock **50**. |
| **Step 2** | Infrastructure | **Pauses** the MongoDB container to simulate a sudden connectivity loss. |
| **Step 3** | Kafka Producer | Produces a valid order message directly to `orders-topic`. |
| **Step 4** | Observation | Waits for 5 seconds. The consumer should be in a retry loop during this time. |
| **Step 5** | Infrastructure | **Unpauses** the MongoDB container to restore connectivity. |
| **Step 6** | State Verification | Polls MongoDB for the order record. Verifies it eventually reaches `PROCESSED` status. |
| **Step 7** | Stock Verification | Asserts that the product stock is correctly decremented (**50** → **47**). |

---

### 🛡️ Expected Resilience Behavior

| Feature | Behavior |
|---------|----------|
| **Retry Logic** | The consumer detects `ServerSelectionTimeoutError` (or equivalent) and enters a retry loop. |
| **No Data Loss** | The consumer does **not** commit the Kafka offset while the database is down. |
| **Self-Healing** | Once the database is reachable, the consumer automatically completes the pending operation. |

---

> 💡 This test confirms that the system can survive transient infrastructure failures without requiring manual intervention or losing business data.
