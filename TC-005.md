# Producer Validation - Invalid Requests

## Implementation Details

---

### 🔧 Pre-conditions

- System is running with Kafka and MongoDB containers.
- Producer API is accessible at the configured URI.

---

### 🧪 Test Steps

| Step | Area | Description |
|------|------|-------------|
| **Step 1** | API Call | Sends a `POST` request to `/api/orders` with `quantity: -5`. |
| **Step 2** | Verification | Asserts that the API returns **400 Bad Request**. |
| **Step 3** | API Call | Sends a `POST` request to `/api/orders` with missing `productId`. |
| **Step 4** | Verification | Asserts that the API returns **400 Bad Request**. |
| **Step 5** | Kafka Check | (Implicit) Verifies that no messages were published for these invalid requests by checking the pipeline state. |

---

### 🛡️ Expected Gatekeeper Behavior

| Feature | Behavior |
|---------|----------|
| **Schema Validation** | The Producer service validates the request payload against the expected schema. |
| **Data Integrity** | Invalid data is rejected at the entry point, preventing pollution of the Kafka stream. |
| **User Feedback** | Clear 4xx error codes are returned to the client to indicate a malformed request. |

---

> 💡 This test ensures the Producer acts as a robust gatekeeper, maintaining the quality of events in the streaming system.
