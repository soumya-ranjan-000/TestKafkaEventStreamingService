# Consumer Group - Load Balancing

## Implementation Details

---

### 🔧 Pre-conditions

- `orders-topic` is created with **3 partitions** (via Kafka AdminClient).
- 3 instances of the `KafkaConsumer` service are running in the same `group_id`.

---

### 🧪 Test Steps

| Step | Area | Description |
|------|------|-------------|
| **Step 1** | Scale Up | Starts 2 additional consumer instances using `TestContainerManager.startAdditionalConsumers(2)`. |
| **Step 2** | Rebalance | Waits for the Kafka consumer group to rebalance partitions among the 3 instances. |
| **Step 3** | Load Injection | Produces **30 unique order messages** via the Producer API. |
| **Step 4** | Observation | Waits for async processing to complete. |
| **Step 5** | Verification | Queries MongoDB to confirm all **30 orders** are processed. |
| **Step 6** | Distribution | (Optional) Queries each consumer instance's `/api/stats` to verify load distribution. |

---

### 🛡️ Expected Scaling Behavior

| Feature | Behavior |
|---------|----------|
| **Parallelism** | Kafka partitions are distributed across the available consumers in the group. |
| **Throughput** | Multiple consumers process messages simultaneously, increasing total system capacity. |
| **No Duplication** | Each message is assigned to exactly one consumer instance within the same group. |

---

> 💡 This test verifies the horizontal scalability of the consumer layer, ensuring the system can handle increased load by adding more instances.
