# Error Handling - Malformed JSON Payload

## Implementation Details

---

### 🔧 Pre-conditions

- System is running with Kafka and MongoDB containers.
- `orders-topic` and `orders-dlq-topic` are accessible.

---

### 🧪 Test Steps

| Step | Area | Description |
|------|------|-------------|
| **Step 1** | Kafka Producer | Manually produces a non-JSON string (e.g., `invalid-data-<timestamp>`) to the `orders-topic` using `KafkaProducerClient`. |
| **Step 2** | DLQ Verification | Polls the `orders-dlq-topic` for the exact malformed string. Verifies it arrives in the DLQ and includes error headers. |
| **Step 3** | Resilience Check | Sends a valid order via the API to confirm the consumer has not crashed and continues to process messages. |
| **Step 4** | State Verification | Verifies the valid order is processed and saved to MongoDB, ensuring no pipeline blockage. |

---

### 🛡️ Expected Resilience Behavior

| Feature | Behavior |
|---------|----------|
| **Fail-Fast** | The consumer detects unparseable JSON immediately and stops trying to map it to an object. |
| **Isolate** | The bad message is routed to the DLQ instead of causing the consumer to loop or crash. |
| **Recovery** | The offset is committed, allowing the consumer to process the next valid message in the queue. |

---

> 💡 This test ensures the system is robust against data corruption and prevents a single bad message from poisoning the entire stream.
